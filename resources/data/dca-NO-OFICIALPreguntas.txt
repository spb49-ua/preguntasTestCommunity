¿Qué es el Diseño por Contrato (DbC)?
3
- Una metodología para diseñar interfaces gráficas.
- Un sistema para validar entradas y salidas de programas.
- Una metodología de diseño de software que aplica conceptos de contratos de negocios.
- Un enfoque orientado exclusivamente a depurar errores en tiempo de ejecución.

¿Qué elementos forman parte de un contrato en DbC?
4
- Precondiciones y excepciones.
- Postcondiciones e invariantes de bucles.
- Invariantes de bucles y estados finales.

2

- El compilador del lenguaje usado.

¿Qué ocurre si una postcondición no se cumple?
3
- El fallo es del cliente que llamó al método.
- El error debe corregirse en tiempo de compilación.
- Es responsabilidad del método que la definió.
- No se genera ningún error, ya que es opcional.

¿Qué son las invariantes de clase?
4
- Condiciones que se cumplen antes de ejecutar un método específico.
- Condiciones que deben cumplir los datos de entrada de un método.
- Propiedades asociadas a variables estáticas de una clase.
- Propiedades lógicas que deben cumplirse antes y después de ejecutar cualquier método de una clase.

¿Qué es un Sistema de Control de Versiones (SCV)?
4
- Un software que permite conocer las modificaciones hechas a un proyecto.
- Un sistema para mantener únicamente las últimas versiones de los archivos.
- Una herramienta exclusiva para revertir cambios recientes.
- Ninguna de las anteriores.

¿Cuál es una función principal de un SCV?
3
- Restringir los cambios en un proyecto a un único desarrollador.
- Mantener siempre la última versión disponible sin posibilidad de revertir cambios.
- Gestionar automáticamente los cambios realizados en uno o varios ficheros de un proyecto.
- Sustituir las copias de seguridad en proyectos colaborativos.

¿Qué nos permite hacer un SCV además de gestionar cambios?
2
- Limitar el acceso al histórico del proyecto.
- Restaurar cualquier archivo del proyecto a un estado anterior, no necesariamente el último.
- Evitar que múltiples programadores trabajen en un mismo proyecto.
- Aislar cambios sin compartirlos con otros desarrolladores.

¿Qué es un repositorio en un SCV?
1
- La copia maestra donde se guardan todas las versiones de los archivos de un proyecto.
- Un directorio local que contiene solo las copias de trabajo.
- Una copia remota de los archivos modificados que aún no se han comprometido.
- Un sistema exclusivo para archivos ejecutables en proyectos de software.

¿Qué diferencia a un SCV centralizado de uno distribuido?
2
- Los SCV centralizados permiten que todos los usuarios tengan su propio repositorio.
- En un SCV distribuido, cada usuario tiene su propio repositorio local que puede sincronizar con un repositorio central.
- Los SCV centralizados funcionan solo con un equipo administrativo que maneja las copias de trabajo.
- Ninguna de las anteriores.

¿Qué es un “Commit” en un SCV?
4
- Una acción que actualiza automáticamente todos los archivos locales con los cambios remotos.
- El proceso de clonar un repositorio central en la máquina local.
- Un registro automático de los cambios realizado sin intervención del programador.
- La acción de guardar cambios en la copia local del repositorio junto con un mensaje descriptivo.

¿Qué acción permite trasladar los contenidos de la copia local de un repositorio al repositorio maestro?
3
- Clone
- Commit
- Push
- Fetch

¿Qué se entiende por conflicto en un SCV?
2
- La situación en la que un archivo no se encuentra bajo control de versiones.
- La situación en la que dos desarrolladores realizan cambios en la misma región de un fichero, generando discrepancias.
- Un error del sistema al realizar un commit simultáneo desde varios usuarios.
- Una falla en el repositorio remoto durante una operación de pull.

¿Qué es una rama en un SCV?
3
- Un método para guardar copias de seguridad de los archivos.
- Un conjunto de cambios descartados de un proyecto.
- Una línea de evolución del software que puede incluir desarrollos paralelos.
- Un repositorio separado para almacenar versiones estables del proyecto.

¿Qué representa el término master o main en un SCV?
1
- Una rama principal donde se suelen mantener las versiones estables del software.
- Un repositorio local exclusivo de un usuario.
- Un archivo de configuración general de un proyecto.
- Una rama de corto recorrido creada para probar características experimentales.

¿Qué tipo de ramas se suelen emplear para trabajar en una característica o corregir un fallo específico?
4

¿Qué similitudes comparten Git, Mercurial y Bazaar?
1
- Los tres son sistemas de control de versiones distribuidos.
- Los tres usan índices para registrar los cambios antes de un commit.
- Los tres dependen de un repositorio centralizado para funcionar.
- Ninguno de ellos soporta la creación de ramas.

¿Qué diferencia clave hay entre Git y Mercurial en cuanto a la gestión de cambios?
4
- Mercurial usa SHA-256, mientras que Git usa SHA-1.
- Git no soporta extensiones, pero Mercurial sí.
- Git organiza el historial como un árbol, mientras que Mercurial lo trata como un grafo.
- Mercurial no requiere usar un índice (stage) antes de un commit, mientras que Git sí.

¿Qué característica especial destaca de Bazaar frente a Git y Mercurial?
3
- Es más rápido que Git y Mercurial.
- Ofrece compatibilidad total con proyectos gestionados por Mercurial.
- Es más fácil de usar y tiene un flujo de trabajo más sencillo.
- Implementa un sistema híbrido centralizado y distribuido.

¿Qué característica define a CVS?
2
- Es un sistema de control de versiones distribuido.
- Es un sistema de control de versiones centralizado.
- Fue diseñado para reemplazar a Git.

¿Qué es una “topic branch”?
3
- Una rama que contiene las versiones estables de un proyecto.
- Una rama que incluye todas las propuestas de cambios realizadas por varios desarrolladores.
- Una rama puntual creada para trabajar en un único aspecto, como una característica o corrección específica.
- Una rama experimental para probar integraciones avanzadas.

¿Qué diferencia clave hay entre merge y rebase en Git?
1
- El merge combina los cambios de dos ramas y preserva el historial completo, mientras que el rebase reescribe el historial para obtener una línea más lineal.
- El rebase es más rápido que el merge, pero no permite resolver conflictos.
- El merge descarta los commits antiguos, mientras que el rebase los mantiene en paralelo.
- El rebase solo funciona en repositorios centralizados.

¿Qué son los submódulos en Git?
2
- Una característica para trabajar con múltiples ramas en paralelo.
- Una forma de incluir otro repositorio Git como parte de un proyecto actual.
- Una alternativa a las herramientas gráficas para manejar ramas.
- Un método para restaurar archivos borrados accidentalmente.

¿Qué diferencia clave tiene Breezy con otros SCV?
3
- Es un SCV centralizado exclusivo para sistemas Unix.
- No soporta operaciones de merge o rebase.
- Es una reimplementación de Bazaar con soporte adicional para funcionalidades modernas.
- Funciona exclusivamente como cliente de Git.

¿Qué es Tailor en el contexto de los SCV?
4
- Un comando avanzado de Git para ajustar ramas.
- Un sistema de control de versiones distribuido diseñado para proyectos pequeños.
- Una extensión de Mercurial para manejar submódulos.
- Una herramienta que permite migrar proyectos entre diferentes sistemas de control de versiones.

¿Qué permite hacer git-annex?
2
- Sincronizar ramas locales con remotas automáticamente.
- Almacenar grandes archivos en repositorios Git sin comprometer el rendimiento.
- Gestionar extensiones de terceros en Git.
- Realizar operaciones avanzadas con submódulos.
