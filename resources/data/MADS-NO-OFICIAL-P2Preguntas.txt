El acoplamiento en diseño de software se refiere a:
2
- El nivel de cohesión entre los módulos.
- La dependencia entre módulos o componentes del sistema.
- La cantidad de funcionalidad en un módulo.
- La capacidad de extender las clases en el sistema.

¿Cuál de las siguientes afirmaciones describe un alto acoplamiento?
3
- Los módulos de un sistema son independientes entre sí.
- El cambio en un módulo no afecta a los demás.
- Un cambio en un módulo provoca modificaciones en varios otros.
- Los módulos comparten interfaces genéricas.

¿Cuál es una consecuencia directa de un diseño con alto acoplamiento?
4
- Aumenta la cohesión entre módulos.
- Se facilita la reutilización del código.
- Mejora la legibilidad del sistema.
- Dificulta realizar cambios o pruebas aisladas.

Un módulo con alta cohesión:
1
- Contiene elementos que están relacionados entre sí y se centran en una única tarea.
- Está estrechamente acoplado con otros módulos.
- Tiene dependencias externas que lo hacen más flexible.
- Puede cambiar por múltiples razones.

¿Cuál es un síntoma de alta cohesión en un módulo?
2
- Los cambios en el módulo afectan a otros módulos.
- Las modificaciones suelen limitarse al interior del módulo.
- El módulo gestiona múltiples responsabilidades.
- Requiere cambios frecuentes en su interfaz.

¿Qué ocurre cuando un sistema tiene módulos altamente cohesivos?
3
- Aumenta la dependencia entre módulos.
- Las funcionalidades se mezclan en varios módulos.
- Los cambios se producen principalmente dentro de los módulos afectados.
- Los módulos se vuelven rígidos ante el cambio.

¿Cuál de los siguientes es un síntoma de un diseño rígido?
2
- Los cambios en el código no introducen errores.
- Las dependencias hacen que los cambios requieran modificaciones en cascada.
- Los módulos son independientes y fáciles de modificar.
- El código es altamente cohesivo.

¿Qué significa que un diseño sea frágil?
3
- Es difícil reutilizar el código.
- Los cambios son fáciles de rastrear y solucionar.
- Cambios en el código provocan errores inesperados en ejecución.
- Las interfaces son estables y bien definidas.

Cuando un diseño es inmóvil, se refiere a:
4
- No permite refactorización.
- Tiene dependencias implícitas.
- Fomenta la herencia múltiple.
- Es difícil reutilizar su código en otros contextos.

¿Qué describe mejor la viscosidad en un mal diseño?
1
- Los cambios correctos son más difíciles de aplicar que soluciones rápidas y erróneas.
- Las clases tienen demasiadas dependencias externas.
- Es fácil realizar cambios pequeños en el diseño.
- La refactorización se realiza sin complicaciones.

El principio de responsabilidad única establece que:
4
- Una clase debe ser capaz de manejar múltiples responsabilidades si son relacionadas.
- Un módulo debe depender solo de otros módulos cohesivos.
- Las interfaces deben ser pequeñas y específicas.
- Una clase debe cambiar por una única razón.

¿Cuál de los siguientes ejemplos viola el principio de responsabilidad única?
1
- Una clase Rectángulo que maneja cálculos geométricos y representación gráfica.
- Una clase Post que almacena texto y autor de un mensaje.
- Una clase Usuario que valida credenciales.
- Una clase Logger que registra eventos en un archivo.

¿Qué implica un diseño que sigue el principio de responsabilidad única?
2
- Los módulos tienen dependencias explícitas con otros módulos.
- Cada clase tiene una tarea bien definida y aislada.
- Las clases tienden a ser más grandes y manejan varias responsabilidades.
- Las interfaces públicas cambian frecuentemente.

¿Cuál de los siguientes diseños aplica correctamente el principio SRP?
3
- Un módulo que agrupa múltiples funcionalidades independientes para mayor eficiencia.
- Una clase con múltiples métodos relacionados con tareas de negocio y de interfaz gráfica.
- Una clase que gestiona únicamente cálculos geométricos, como el área de figuras.
- Un sistema que basa su diseño en clases altamente acopladas.

El principio Open-Closed establece que:
3
- Las clases deben depender de abstracciones, no de implementaciones concretas.
- Las interfaces deben ser pequeñas y específicas para cada cliente.
- Las clases deben estar abiertas para extensión pero cerradas para modificación.
- Las clases base no deben tener métodos abstractos.

¿Qué técnica se usa comúnmente para implementar el principio Open-Closed?
2
- Modificar las clases base para añadir nuevas funcionalidades.
- Extender clases mediante herencia o composición.
- Reducir el número de interfaces en el diseño.
- Limitar el uso de interfaces públicas.

Un ejemplo de mal uso del principio Open-Closed es:
4
- Crear una nueva subclase para añadir comportamiento adicional.
- Usar interfaces para abstraer funcionalidades.
- Implementar una clase derivada que amplía las capacidades de una clase base.
- Cambiar directamente el código de una clase ya utilizada en producción.

¿Cuál de las siguientes afirmaciones describe correctamente un diseño que sigue OCP?
1
- Se pueden añadir funcionalidades nuevas sin modificar el código existente.
- Se evita completamente el uso de herencia en el diseño.
- Todas las dependencias del sistema están basadas en implementaciones concretas.
- Las clases no tienen interfaces públicas.

El principio de sustitución de Liskov establece que:
1
- Una clase derivada debe ser sustituible por su clase base sin alterar el comportamiento del programa.
- Una clase base no debe depender de clases derivadas.
- Las interfaces deben dividirse en varias más específicas.
- Los módulos de alto nivel no deben depender de detalles.

¿Cuál de los siguientes ejemplos viola el principio de sustitución de Liskov?
2
- Un Mamífero como clase base de Gato y Caballo.
- Un Rectángulo como clase base de un Cuadrado.
- Una clase Figura extendida por Triángulo y Círculo.
- Una interfaz genérica implementada por varias clases concretas.

¿Qué condición debe cumplir una clase derivada para seguir el principio LSP?
4
- Sobrescribir todos los métodos de la clase base.
- Definir nuevos métodos que no están en la clase base.
- Cambiar el comportamiento de los métodos base si es necesario.
- Preservar el comportamiento esperado de la clase base en todos los contextos.

Un ejemplo de diseño que sigue el principio LSP es:
3
- Un Coche que hereda de Avión para reutilizar métodos.
- Un Cuadrado que sobrescribe métodos de un Rectángulo para mantener igualdad en ancho y alto.
- Un Mamífero que sirve como base para un Perro y un Gato.
- Una clase derivada que ignora completamente las restricciones de la clase base.

El principio de segregación de interfaces establece que:
2
- Las clases deben ser pequeñas y cohesivas.
- Los clientes no deben depender de interfaces que no usan.
- Las dependencias deben basarse en implementaciones concretas.
- Los métodos de las interfaces deben ser abstractos.

¿Cuál de los siguientes ejemplos aplica el principio ISP?
4
- Una interfaz única con métodos para lectura y escritura.
- Una clase que implementa una interfaz con métodos que no utiliza.
- Una interfaz genérica que obliga a los clientes a implementar métodos innecesarios.
- Dos interfaces pequeñas y específicas para lectura y escritura, implementadas por clases separadas.

¿Qué problema resuelve el principio ISP?
1
- La dependencia innecesaria de los clientes en métodos que no utilizan.
- El acoplamiento excesivo entre módulos del sistema.
- La falta de cohesión en las clases.
- La rigidez en la estructura jerárquica del código.

El principio de inversión de dependencias establece que:
3
- Los módulos de bajo nivel deben depender de módulos de alto nivel.
- Las clases concretas deben depender de abstracciones.
- Tanto los módulos de alto como de bajo nivel deben depender de abstracciones.
- Las interfaces deben estar fuertemente relacionadas con las implementaciones.

¿Cuál de los siguientes diseños sigue el principio DIP?
2
- Una clase Botón que controla directamente una Lámpara específica.
- Un Botón que utiliza una interfaz Switchable para controlar cualquier dispositivo.
- Una clase Controlador que depende directamente de las clases concretas que gestiona.
- Un módulo de bajo nivel que accede directamente a un módulo de alto nivel.

¿Qué ventaja ofrece seguir el principio DIP en un diseño?
4
- Se minimiza la cantidad de código en las clases base.
- Se simplifica la herencia entre clases.
- Se eliminan completamente las dependencias en el sistema.
- Se facilita el reemplazo de implementaciones concretas sin modificar el código de alto nivel.
